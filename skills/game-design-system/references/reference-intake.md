# 레퍼런스 인테이크

요청이 스크린샷, URL, 무드보드, 경쟁작, 아트 디렉션 노트, 덜 정리된 형용사 같은 느슨한 영감 자료로 시작될 때 이 파일을 사용한다.

## 인테이크 체크리스트

- 제품: 게임명 또는 작업 코드명, 타깃 유저, 장르, 카메라 방식, 세션 길이.
- 플랫폼: PC, 콘솔, 모바일, 핸드헬드, 웹, TV, 크로스플랫폼.
- 입력 모델: 마우스/키보드, 컨트롤러, 터치, 혼합 입력.
- UI 범위: 게임플레이 HUD, 메타 UI, 또는 둘 다.
- 비즈니스 제약: F2P, 패키지형, 라이브옵스, 시즌 이벤트, 상점 요구사항.
- 산출물 목표: 컨셉 방향 제안, 실무 투입 가능한 명세, 기존 시스템 점검/리디자인.
- 기술 타깃: Figma 전용, 프론트엔드 토큰, Unity, Unreal, 커스텀 엔진.

## 증거 유형

- 이미지 또는 스크린샷: 계층, 밀도, 형태 언어, 대비, 채도 운용, 아이콘 스타일, 구성 분석에 유리하다.
- URL 또는 라이브 페이지: 흐름, 상호작용 리듬, 반응형 상태, 내비게이션 구조를 보기 좋다.
- 텍스트 설명: 의도, 비즈니스 목표, 무드, 제약, 비목표를 파악하기 좋다.
- 기존 제품 UI: 연속성 제약과 마이그레이션 리스크를 파악하기 좋다.

## 추출 매트릭스

각 레퍼런스마다 다음 항목을 기록한다.

- `Signal`: 직접 보이거나 명시적으로 적혀 있는 사실
- `Inference`: 그 사실이 시스템에 시사하는 바에 대한 해석
- `Adopt`: 정식 규칙으로 채택할 것
- `Adapt`: 대상 게임에 맞게 변형할 것
- `Reject`: 레퍼런스 수준에만 남기고 버릴 것

추출 시에는 다음 범주를 사용한다.

- 무드: heroic, tactical, cozy, oppressive, ceremonial, futuristic, retro 등
- 계층: 무엇이 크게 보여야 하는지, 무엇이 배경처럼 남아야 하는지, 무엇이 조작 가능해야 하는지
- 밀도: 여백이 많은 시네마틱 UI인지, 데이터가 많은 관리형 UI인지
- 형태 언어: 날카로운지, 베벨 처리됐는지, 둥근지, 카드형인지, 분절형인지, 다이에제틱한지
- 색상 운용: 액센트 절제, 의미 색상, 희귀도 색상, 표면 분리 방식
- 타이포그래피: 디스플레이형 vs 유틸리티형, 숫자 강조, 라벨 리듬, 현지화 내성
- 어포던스: 조작 가능한 요소가 정보용 요소와 어떻게 구분되는지
- 모션: 절제된 전환 중심인지, 강한 피드백 중심인지
- 플랫폼 단서: 거실 시청 거리, 핸드헬드 가독성, 터치 목표 크기, 모니터 멀티태스킹

## 충돌 해결

레퍼런스가 서로 다른 방향을 가리키면 아래 순서로 우선순위를 정한다.

1. 게임플레이 가독성
2. 입력 명확성
3. 플랫폼 적합성
4. 콘텐츠 밀도 요구
5. 시각 정체성
6. 새로움

많이 보이는 레퍼런스 선택을 뒤집었다면 이유를 설명한다.

## 누락 정보

빠진 정보가 핵심 시스템 결정을 바꿀 때만 추가 질문을 한다. 예시는 다음과 같다.

- 게임이 컨트롤러 우선인지 터치 우선인지
- 전투 중 HUD가 항상 표시돼야 하는지
- 수익화 또는 라이브 이벤트 화면이 범위에 포함되는지
- 현지화 비중이 높은 언어를 지원해야 하는지

안전하게 가정할 수 있는 정보 때문에 진행을 멈추지 않는다. 가정을 명시하고 계속 진행한다.
