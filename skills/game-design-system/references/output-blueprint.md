# 출력 청사진

최종 응답이나 명세 문서를 구성할 때 이 파일을 사용한다.

## 권장 순서

1. 시스템 논지
2. 제품 맥락과 가정
3. 레퍼런스 분석
4. 디자인 원칙
5. 기반 토큰
6. 화면 맵
7. 컴포넌트 명세
8. 상호작용과 모션 규칙
9. 접근성과 현지화 메모
10. 구현 핸드오프

## 섹션 가이드

### 1. 시스템 논지

플레이어가 느낄 인상과 시스템의 운영 목적을 2-4문장으로 먼저 설명한다.

### 2. 제품 맥락과 가정

디자인에 실제 영향을 주는 가정만 적는다.

- 플랫폼과 입력 모델
- 장르와 게임 속도
- UI 범위
- 수익화 또는 라이브옵스 필요
- 카메라 거리 또는 장면 복잡도

### 3. 레퍼런스 분석

간결한 증거 표를 사용한다.

| 출처 | 유지 | 변형 | 제외 | 메모 |
| --- | --- | --- | --- | --- |

### 4. 디자인 원칙

각 원칙에는 다음 항목을 포함한다.

- 규칙
- 근거
- 실제 결과

### 5. 기반 토큰

작은 표를 사용한다. 권장 토큰 그룹은 다음과 같다.

- 색상 역할
- 타이포그래피 역할
- 간격과 크기
- 반경과 스트로크
- 입체감과 오버레이
- 모션 타이밍과 이징

### 6. 화면 맵

게임플레이, 메타, 경제, 소셜/시스템 화면 기준으로 시스템을 묶는다.

### 7. 컴포넌트 명세

우선순위가 높은 컴포넌트마다 다음을 포함한다.

- 목적
- 계층
- 변형
- 상태
- 콘텐츠 규칙
- 플랫폼/입력 메모

### 8. 상호작용과 모션 규칙

시스템이 다음 상황에 어떻게 반응하는지 설명한다.

- focus와 hover
- confirm과 deny
- 보상과 성장 순간
- 오류와 경고 상태
- 전환 속도감

### 9. 접근성과 현지화 메모

다음 항목을 다룬다.

- 색상 외 중복 단서
- 글자 크기 조절과 텍스트 확장
- CJK와 긴 문자열 대응력
- 컨트롤러 포커스 명확성
- 필요한 경우 터치 목표 크기

### 10. 구현 핸드오프

마지막에는 실행 계층을 정리한다.

- Figma 변수, 스타일, 컴포넌트 세트
- 토큰 네이밍 규칙
- Unity 또는 Unreal 매핑
- 반응형 또는 브레이크포인트 규칙
- 프로토타입 검증이 필요한 부분
